# Retrieval Layer

## Architecture

The Day-1 retrieval layer has three small parts:

1. `src/build_index.py`
   - Reads `data/product_catalog.csv` and `data/faq.csv`.
   - Converts each row into one searchable document.
   - Builds two ChromaDB collections: `catalog` and `faq`.

2. `src/vector_store.py`
   - Opens the persisted Chroma index.
   - Provides `search(query, collection, top_k)`.
   - Returns `content`, `metadata`, and similarity `score`.

3. `src/embeddings.py`
   - Uses local `sentence-transformers/all-MiniLM-L6-v2` embeddings.
   - Can use deterministic local hash embeddings only for basic smoke tests.

This keeps the retrieval layer replaceable. Later, the chat pipeline only needs to call `vector_store.search()`.

## Document Schema

Catalog document metadata:

```python
{
    "source": "product_catalog.csv",
    "doc_type": "catalog",
    "product_id": "P002",
    "name": "Men's Linen Shirt",
    "variant": "M / White",
    "category": "Men's Tops",
    "price": 185000,
    "stock": 8,
}
```

FAQ document metadata:

```python
{
    "source": "faq.csv",
    "doc_type": "faq",
    "faq_id": "F007",
    "category": "shipping",
    "question": "Can I get same-day delivery?",
}
```

## Chunking Strategy

For this mock dataset, use one row as one chunk.

Catalog:
- One product variant equals one document.
- Variant, price, stock, category, and description stay together.
- This avoids retrieving a product name without the exact stock or price.

FAQ:
- One FAQ row equals one document.
- Question and answer stay together.
- This avoids answering from only a partial policy.

When real datasets become larger, only split long policy pages. Keep product SKUs/variants atomic.

## Recommendations

Embedding model:
- Primary: `sentence-transformers/all-MiniLM-L6-v2`
- Reason: local semantic embeddings keep retrieval independent from OpenAI while Groq Llama handles response generation.

LLM:
- Primary: `llama-3.1-8b-instant` via Groq.
- Note: this is a chat model, not an embedding model.

Vector database:
- Primary: ChromaDB local persistence.
- Reason: no hosted infrastructure, easy demo, easy replacement later.

## Indexing Pipeline

```text
CSV rows -> retrieval documents -> embeddings -> Chroma collections
```

Run:

```bash
python src/build_index.py
```

Re-run this whenever `data/product_catalog.csv` or `data/faq.csv` changes.

## Retrieval Pipeline

```python
from src.vector_store import search

matches = search(
    query="do you have same day delivery with gosend?",
    collection="faq",
    top_k=3,
)
```

Expected result shape:

```python
[
    {
        "content": "FAQ topic: shipping\nQuestion: ...",
        "metadata": {"faq_id": "F007", "category": "shipping", ...},
        "score": 0.83,
    }
]
```

## Test Retrieval Quality

After building the index, run:

```bash
python src/evaluate_retrieval.py
```

The script checks mock queries against expected top results:
- `P002` for men's linen shirt M white.
- `F007` for same-day GoSend delivery.
- `F003` for payment methods.
- `P015` for black canvas tote bag.

If a result is marked `CHECK`, inspect whether the correct document is still in the top 3. For small mock data, top-1 should usually be correct, but semantically similar product variants can be close.

## Common Pitfalls

- Rebuilding with one embedding model and querying with another. The wrapper detects this and asks you to rebuild.
- Forgetting to rebuild after CSV changes. Chroma will still contain old data until `python src/build_index.py` is rerun.
- Chunking product name separately from variant, price, or stock. That makes the bot more likely to mix facts.
- Mixing catalog and FAQ into one collection too early. Separate collections make intent routing and source audit easier.
- Treating local hash embeddings as production quality. They are only a no-key fallback for pipeline testing.
- Trying to use `llama-3.1-8b-instant` as the vector embedding model. It should generate answers after retrieval, not build the vector index.
